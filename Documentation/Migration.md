# Migration Guide — Integrations & Addons

This guide is for addon authors who previously integrated with Yapper via
direct internal calls (e.g. calling into `YapperTable.*` or hooking core
functions). The public API (`_G.YapperAPI`) is the stable, supported
integration surface. Direct calls into internal modules are unsupported and
may break without notice.

For a full API reference see [API.md](API.md). For writing a new bridge from
scratch see [Bridges.md](Bridges.md).

---

## 1. Guarding for the API

`Src/API.lua` is loaded early (after `Src/Events.lua`), but addon load order
can vary. Always guard your registration:

```lua
local function RegisterWithYapper()
    local api = _G.YapperAPI
    if not api then return end

    api:RegisterCallback("EDITBOX_SHOW", function(chatType, target)
        -- ...
    end)
end

local f = CreateFrame("Frame")
f:RegisterEvent("PLAYER_ENTERING_WORLD")
f:SetScript("OnEvent", function(self)
    self:UnregisterEvent("PLAYER_ENTERING_WORLD")
    RegisterWithYapper()
end)
```

## 2. Common Migration Patterns

### Typing indicators (was: direct bridge calls)

**Before** (internal — do not use):
```lua
YapperTable.TypingTrackerBridge:OnOverlayFocusGained(chatType)
YapperTable.TypingTrackerBridge:OnOverlayFocusLost()
```

**After** (public API):
```lua
local api = _G.YapperAPI

api:RegisterCallback("EDITBOX_SHOW", function(chatType, target)
    MyAddon:OnTypingStarted(chatType)
end)

api:RegisterCallback("EDITBOX_HIDE", function()
    MyAddon:OnTypingStopped()
end)

api:RegisterCallback("EDITBOX_CHANNEL_CHANGED", function(newType, newTarget)
    MyAddon:OnChannelChanged(newType, newTarget)
end)
```

---

### Text transformation before send (was: Chat hook or PRE_SEND direct call)

Use `PRE_SEND` to modify the full text before it reaches the chunker. Only
the first chunk will carry any prefix you add here.

```lua
_G.YapperAPI:RegisterFilter("PRE_SEND", function(p)
    if MyAddon:ShouldApplyPrefix(p.chatType) then
        p.text = MyAddon:GetPrefix() .. p.text
    end
    return p  -- return nil or false to cancel the send entirely
end, 20)  -- priority 20; lower numbers run first (default 10)
```

**Filter payload fields** (`p`):

| Field | Type | Description |
| :--- | :--- | :--- |
| `text` | string | The full outgoing message. Mutate to transform it. |
| `chatType` | string | `"SAY"`, `"PARTY"`, `"WHISPER"`, etc. |
| `target` | string | Recipient name (whispers/BNet) or channel number. |
| `language` | string | Language passed to `SendChatMessage`. |

---

### Suppressing the Yapper overlay (was: WIM-style direct hook)

Return `false` from `PRE_EDITBOX_SHOW` to prevent the overlay from opening.
This is how `WIMBridge` suppresses Yapper when WIM owns keyboard focus.

```lua
_G.YapperAPI:RegisterFilter("PRE_EDITBOX_SHOW", function(p)
    if MyAddon:OwnsFocus() then
        return false  -- cancel overlay open
    end
    return p
end, 5)  -- low priority so it runs before general-purpose filters
```

---

### Reacting to a completed send

```lua
_G.YapperAPI:RegisterCallback("POST_SEND", function(text, chatType, target)
    MyAddon:OnMessageSent(text, chatType, target)
end)
```

---

### Claiming a post (delegated delivery)

If your addon handles its own message delivery (e.g. via LibGopher), register
for `POST_CLAIMED` to receive chunks Yapper has delegated:

```lua
_G.YapperAPI:RegisterCallback("POST_CLAIMED", function(handle)
    -- handle.text, handle.chatType, handle.target are available
    MyAddon:Deliver(handle)
    -- call ResolvePost when delivery is confirmed:
    _G.YapperAPI:ResolvePost(handle)
end)
```

## 3. Filter Priority Reference

| Priority range | Intended use |
| :--- | :--- |
| 1 – 9 | Security / suppression filters (run first; can cancel the pipeline) |
| 10 – 19 | Standard consumer transforms (default) |
| 20 – 29 | Post-consumer transforms (e.g. prefix injection after other processing) |
| 30+ | Last-resort / logging filters |

## 4. All Filter & Callback Events

See [API.md](API.md) for the full list of filter hookpoints and callback
events. A summary of the most commonly used:

| Filter | When it fires |
| :--- | :--- |
| `PRE_EDITBOX_SHOW` | Before the overlay opens |
| `PRE_SEND` | Before the text is chunked or queued |
| `PRE_CHUNK` | On each individual chunk before it enters the queue |
| `PRE_SPELLCHECK` | Before a word is looked up in the dictionary |
| `PRE_DELIVER` | Just before each chunk is handed to the WoW API |

| Callback | When it fires |
| :--- | :--- |
| `EDITBOX_SHOW` | Overlay opened |
| `EDITBOX_HIDE` | Overlay closed |
| `EDITBOX_CHANNEL_CHANGED` | Channel/target changed while overlay is open |
| `POST_SEND` | Full message send initiated |
| `POST_CLAIMED` | Chunk delegated for external delivery |
| `CONFIG_CHANGED` | A config value changed |
| `THEME_CHANGED` | Active theme changed |
| `SPELLCHECK_SUGGESTION` | Suggestion popup shown for a word |
| `SPELLCHECK_APPLIED` | User accepted a suggestion |

## 5. Avoiding Double-Calls

If your addon embeds or forks Yapper's bridge files, remove any inlined
direct callsites (e.g. `YapperTable.SomeBridge:X()`) and replace them with
API registrations. Keeping both will cause duplicate signals.

---

## 6. Testing Checklist

- [ ] Guard `_G.YapperAPI` — addon works when Yapper is disabled or not present.
- [ ] `PRE_SEND` / `PRE_DELIVER` filters return `p` (modified or unmodified), not `nil`, unless intentionally cancelling.
- [ ] No calls into `YapperTable.*` internal modules remain.
- [ ] Tested in combat lockdown — no taint from secure frame interactions.
- [ ] Tested in instance and open-world SAY/YELL (different Queue policies apply).
- [ ] Tested with BNet whispers and CLUB channels if your bridge touches those types.
