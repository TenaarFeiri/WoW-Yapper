# Compatibility Bridges

Yapper includes dedicated bridge modules to ensure it plays nicely with other popular chat and RP addons. These are located in `Src/Bridges/`.

## LibGopher / CrossRP (`GopherBridge.lua`)

LibGopher is the standard post-splitting library used by many RP addons (like CrossRP). Since Yapper implements its own splitting, a conflict normally occurs. The `GopherBridge` resolves this by:
1.  **Detection**: Identifying if `LibGopher` is present in the environment.
2.  **Redirection**: If active, Yapper bypasses its own `Queue.lua` and forwards pre-split chunks directly to `LibGopher`.
3.  **State Management**: Monitoring if Gopher's internal queue is busy and suppressing Yapper's overlay if so.

### `YapperTable.GopherBridge` API
- `IsActive()`: Returns true if the bridge is enabled and LibGopher is found.
- `IsBusy()`: Returns true if LibGopher has messages currently queueing.
- `Send(msg, type, ...)`: Wraps a message for LibGopher consumption.

## Simply Typing - Typing Tracker (`TypingTrackerBridge.lua`)

This bridge allows Yapper's custom overlay to trigger typing indicators in standard RP addons that support the "Simply Typing" protocol.

### Mechanics
- **Focus Tracking**: When the Yapper overlay gains focus, the bridge sends a "START_TYPING" signal via `C_ChatInfo.SendAddonMessage`.
- **Channel Switching**: If you change channels (e.g., from `/s` to `/w`) while typing, the bridge immediately updates the signal so other players see where you are targeting.
- **Auto-Stop**: Signals are automatically stopped when you press Enter, Escape, or the overlay hides.

### `YapperTable.TypingTrackerBridge` API
- `OnOverlayFocusGained(chatType)`: Triggers the "is typing" ticker.
- `OnOverlayFocusLost()`: Sends the "stopped typing" signal.
- `OnChannelChanged(newType)`: Updates the current target channel.

## WIM (WoW Instant Messenger)

WIM creates its own whisper edit boxes and handles whisper focus independently. When WIM has an active whisper window with focus, Yapper should step back and let WIM drive the keyboard.

The `WIMBridge` centralises all WIM detection into a single bridge module (`Src/Bridges/WIMBridge.lua`) and uses the public API filter system to suppress the overlay:

1. **Detection**: Checks `_G.WIM.EditBoxInFocus` for `IsShown`, `IsVisible`, or `HasFocus`.
2. **API filter**: Registers a `PRE_EDITBOX_SHOW` filter (priority 5) that returns `false` when WIM owns focus for whisper or unset chat types, cancelling the overlay open.
3. **Direct queries**: Exposes `IsFocusActive()` for internal callers (e.g. the BNet transition guard in `Hooks.lua`) that need a synchronous yes/no answer outside the filter path.

Because the bridge uses the public API, third-party addons can register their own `PRE_EDITBOX_SHOW` filters to participate in overlay-suppression decisions. If WIM is ever removed from the environment, the bridge simply becomes a no-op.

### `YapperTable.WIMBridge` API
- `Init()`: Registers the `PRE_EDITBOX_SHOW` filter via `YapperAPI`. Called from `Chat:Init`.
- `IsLoaded()`: Returns `true` if `_G.WIM` exists.
- `IsFocusActive()`: Returns `true` if a WIM edit box currently has user focus.

## RP Prefix (`RPPrefixBridge.lua`)

RP Prefix (by Zee / Songzee – Argent Dawn EU) lets role-players prepend a short tag (e.g. `[OOC]`) to their `/s` and `/y` messages. Without a bridge, its `C_ChatInfo.SendChatMessage` hook fires on every individual chunk Yapper queues, attaching the prefix to *every* continuation post.

The `RPPrefixBridge` resolves this by:

1. **Detection**: Checking `Zee.RPPrefix` at `Chat:Init` time (during `PLAYER_ENTERING_WORLD`, after all addons are loaded).
2. **Raw-API patching**: Caching the unwrapped `C_ChatInfo.SendChatMessage` (either from `Zee.RPPrefix.SavedSendChatMessage` if the toggle has already fired, or from the global directly) and replacing `Router.SendChatMessage` with it. Per-chunk sends therefore bypass RPPrefix's hook entirely.
3. **First-chunk injection**: Exposing `ApplyPrefix(text, chatType)`, called by `Chat:OnSend` before the message reaches the chunker. When RPPrefix is enabled and all its own conditions are met (`SAY`/`YELL`, not a slash command, prefix not the placeholder), the prefix is prepended to the full text so only the first post carries it.

RPPrefix's toggle button continues to control whether the prefix is active; Yapper simply moves the injection point earlier in the pipeline.

### `YapperTable.RPPrefixBridge` API
- `Init()`: Detects RPPrefix, patches `Router.SendChatMessage`, and activates the bridge. Called automatically from `Chat:Init`.
- `IsActive()`: Returns true if RPPrefix was found and the bridge is enabled.
- `ApplyPrefix(text, chatType)`: Returns the message with the RP prefix prepended when RPPrefix is on and conditions match; otherwise returns `text` unchanged.

## Recent refactor: API integration

As of the refactor, Yapper exposes a public integration surface (`_G.YapperAPI`) and
converts internal bridge wiring to consume that API instead of relying on
hardcoded direct calls from core modules. The goal is to provide a stable,
documented hook set for external addons and to make Yapper's own bridges "eat
their own dog food" by using the public API.

Key changes
* `RPPrefixBridge` now registers a `PRE_SEND` filter (priority 20) that
	applies the prefix to the full outgoing text *before* chunking. This
	guarantees the prefix appears only on the first post.
* `TypingTrackerBridge` registers callbacks for `EDITBOX_SHOW`, `EDITBOX_HIDE`,
	`POST_SEND`, and `EDITBOX_CHANNEL_CHANGED`. EditBox no longer calls the
	bridge directly.
* `GopherBridge` registers for `CONFIG_CHANGED` so it can react to the
	`System.EnableGopherBridge` toggle via the API. (Legacy direct calls in
	config paths remain for a short compatibility window.)

Example — PRE_SEND registration (RPPrefix-style):

```lua
local handle = _G.YapperAPI:RegisterFilter("PRE_SEND", function(p)
		p.text = MyBridge:MaybeApplyPrefix(p.text, p.chatType)
		return p
end, 20)
```

Example — overlay lifecycle callback (TypingTracker-style):

```lua
_G.YapperAPI:RegisterCallback("EDITBOX_SHOW", function(chatType, target)
		TypingTrackerBridge:SignalTyping(chatType)
end)
```

Migration notes for bridge maintainers
- Ensure `_G.YapperAPI` exists before registering — the TOC loads `Src/API.lua`
	early, but addon load order can vary. Guard or delay registrations if needed.
- Prefer the API-driven registrations and remove direct callsites from Yapper's
	core files in forked/embedded bridges to avoid duplicate signals.

See `Documentation/API.md` for the canonical filter and callback reference.
