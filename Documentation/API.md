Yapper Public API
=================

Overview
--------
This document describes the public integration surface exposed by Yapper
via the global `_G.YapperAPI` object. The API provides two complementary
mechanisms:

- Filters — run before an operation (can inspect, modify, or cancel)
- Callbacks — run after an operation (notification only)

The API is intentionally small and stable. All external callbacks are
wrapped in `pcall()` by Yapper so a consumer error cannot crash the addon.

Requirements
------------
- `Src/API.lua` must be loaded before your code registers filters or
  callbacks. The addon TOC places `Src/API.lua` early; if you depend on
  Yapper's API from another addon, ensure you check for `_G.YapperAPI`.

Filters (pre-hooks)
-------------------
Use `YapperAPI:RegisterFilter(hookPoint, callback, priority)`.

- Signature: `function(payload) -> payload|false`
  - Receives a single TABLE (`payload`) with named fields.
  - Return the (possibly modified) `payload` to continue, or `false` to cancel.
  - Returning `nil` is treated as "no change" but avoid relying on it.

Available filter hook points
- `PRE_EDITBOX_SHOW` — payload: `{ chatType, target }` (cancellable — return `false` to suppress the overlay)
  Fires before the overlay opens. Used by WIMBridge to yield focus when WIM owns the whisper window; other addons can register here to suppress the overlay for their own reasons.
- `PRE_SEND` — payload: `{ text, chatType, language, target }` (cancellable)
  Fires after the user presses Enter but before the message is routed. Modify payload fields to rewrite the message or return `false` to block it.
- `PRE_CHUNK` — payload: `{ text, limit }` (cancellable)
  Fires before the chunker splits a long message. Modify `text` or `limit`, or return `false` to prevent chunking entirely.
- `PRE_SPELLCHECK` — payload: `{ text }` (cancellable)
  Fires before the spellchecker runs on the current input. Return `false` to skip spellchecking for this particular text.
- `PRE_DELIVER` — payload: `{ text, chatType, language, target }` (cancellable — triggers delegation)
  Fires in `DirectSend` just before `Router:Send`. Return `false` to claim the message for delegation (see Post Delegation below). The message will not be sent by Yapper; the claiming addon receives a `POST_CLAIMED` callback and must call `YapperAPI:ResolvePost(handle)` within the timeout.

Example — rewrite outgoing text:

```lua
local h = YapperAPI:RegisterFilter("PRE_SEND", function(p)
    p.text = p.text:gsub("brb", "be right back")
    return p
end, 5) -- lower priority number fires earlier

-- Later, to remove:
YapperAPI:UnregisterFilter(h)
```

Example — block messages containing a token:

```lua
YapperAPI:RegisterFilter("PRE_SEND", function(p)
    if p.text:find("FORBIDDEN") then return false end
    return p
end)
```

Callbacks (post-hooks / events)
-------------------------------
Use `YapperAPI:RegisterCallback(event, callback)`.

- Signature: `function(...)` — event-specific arguments are passed.
- These cannot cancel operations; they are notifications only.

Available callback events
- `POST_SEND` — `(text, chatType, language, target)`
  Fires after a message has been handed to the WoW send API.
- `CONFIG_CHANGED` — `(path, value)` where `path` is dot-delimited (e.g. `"EditBox.FontSize"`)
  Fires when a Yapper setting changes.
- `EDITBOX_SHOW` — `(chatType, target)`
  Fires when the Yapper overlay becomes visible.
- `EDITBOX_HIDE` — `()`
  Fires when the Yapper overlay is hidden.
- `EDITBOX_CHANNEL_CHANGED` — `(chatType, target)`
  Fires when the user switches chat channel (Tab, slash command, etc.).
- `THEME_CHANGED` — `(themeName)`
  Fires when the active theme is changed.
- `API_ERROR` — `(kind, hook, handler_info, errorMessage, data, ...)`
  Fires when a filter or callback handler errors. Delivered only to the addon that owns the faulting handler (matched by source file path).
- `SPELLCHECK_SUGGESTION` — `(word, suggestions)`
  Fires when the spellcheck suggestion popup is shown. `suggestions` is an array of word strings.
- `SPELLCHECK_APPLIED` — `(original, replacement)`
  Fires when the user accepts a spellcheck suggestion, replacing `original` with `replacement`.
- `SPELLCHECK_WORD_ADDED` — `(word, locale)`
  Fires when a word is added to the user dictionary (via UI, API, or YALLM auto-learn).
- `SPELLCHECK_WORD_IGNORED` — `(word, locale)`
  Fires when a word is marked as "ignored" in the user dictionary.
- `YALLM_WORD_LEARNED` — `(word, locale)`
  Fires when YALLM auto-promotes a word to the user dictionary after persistent usage (reaching the auto-learn threshold).
- `POST_CLAIMED` — `(handle, text, chatType, language, target)`
  Fires when a `PRE_DELIVER` filter claims a message for delegation. The addon must call `YapperAPI:ResolvePost(handle)` within 5 seconds to confirm it handled delivery (see Post Delegation below).
- `QUEUE_STALL` — `(chatType, policyClass, chunksRemaining)`
  Fires when the ack-event stall timer expires before the server confirmed a chunk. Indicates the Continue prompt is now visible. `policyClass` is the internal policy class string (e.g. `"INSTANCE_LOCAL"`); `chunksRemaining` includes the stalled chunk.
- `QUEUE_COMPLETE` — `()`
  Fires when the delivery queue finishes (all chunks delivered, or queue cancelled). Pair with `QUEUE_STALL` to track queue sessions.

Example — logging sent messages:

```lua
YapperAPI:RegisterCallback("POST_SEND", function(text, chatType, language, target)
    print("Yapper sent:", text, "->", chatType, target)
end)
```

Read-only accessors
-------------------
YapperAPI also exposes a few safe read-only helpers:

- `YapperAPI:GetVersion()` → string (api/addon version)
- `YapperAPI:GetCurrentTheme()` → string | nil
- `YapperAPI:IsOverlayShown()` → boolean
- `YapperAPI:GetConfig(path)` → value (shallow-copies tables)

Queue accessors
---------------
- `YapperAPI:GetQueueState()` → table
  Returns a snapshot of the delivery queue with the following fields:
  - `active` (bool): whether the queue is currently processing
  - `stalled` (bool): whether the Continue prompt is currently shown
  - `chatType` (string|nil): the chat type of the current or next entry
  - `policyClass` (string|nil): the resolved policy class name
  - `pending` (int): chunks waiting in the queue
  - `inFlight` (int): chunks currently sent but not yet ack'd (0 or 1)

- `YapperAPI:CancelQueue()` → int
  Cancels the active queue and discards all pending chunks. Returns the number of chunks discarded. Prints the same chat-frame notice as the built-in double-Escape cancel.

Example — show a custom notice when a queue stalls:

```lua
YapperAPI:RegisterCallback("QUEUE_STALL", function(chatType, policyClass, remaining)
    print("Yapper: waiting for server ack (", remaining, "chunks left)")
end)

YapperAPI:RegisterCallback("QUEUE_COMPLETE", function()
    print("Yapper: queue finished")
end)
```

Example — inspect queue state:

```lua
local state = YapperAPI:GetQueueState()
if state.stalled then
    print("Queue stalled on", state.chatType, "—", state.pending, "chunks pending")
end
```

Utility helpers
---------------
Safe wrappers around `Utils.lua` for bridge and companion-addon authors.
`_G.YAPPER_UTILS` remains the internal/compat surface for early-loading patches;
these API methods are the stable, documented equivalents.

- `YapperAPI:IsChatLockdown()` → boolean
  Returns true if `C_ChatInfo.InChatMessagingLockdown()` is active. Use this to guard sends in bridges the same way Yapper does internally.

- `YapperAPI:IsSecret(value)` → boolean
  Returns true if a value should not be logged or persisted. Uses Blizzard's `issecretvalue`/`canaccessvalue` APIs with a `|K` token fallback.

- `YapperAPI:GetChatParent()` → Frame
  Returns the correct UI parent for chat-related frames, respecting fullscreen panels like the housing editor.

- `YapperAPI:MakeFullscreenAware(frame)`
  Hooks the frame so it re-parents automatically whenever the active fullscreen panel changes. Use this for any Yapper-adjacent UI frame that needs to stay visible when UIParent is hidden.

Example — create a frame that coexists with the housing editor:

```lua
local f = CreateFrame("Frame", nil, YapperAPI:GetChatParent())
YapperAPI:MakeFullscreenAware(f)
```

Example — guard a send against messaging lockdown:

```lua
YapperAPI:RegisterFilter("PRE_SEND", function(p)
    if YapperAPI:IsChatLockdown() then
        return false  -- don't send while locked down
    end
    return p
end)
```

Spellcheck accessors
--------------------
Safe wrappers around the spellcheck system. All return `false` or `nil`
if spellcheck is unavailable or disabled.

- `YapperAPI:IsSpellcheckEnabled()` → boolean
- `YapperAPI:CheckWord(word)` → boolean (true if the word is in the dictionary or user dict)
- `YapperAPI:GetSuggestions(word)` → array of suggestion strings, or nil
- `YapperAPI:GetSpellcheckLocale()` → locale string (e.g. `"enUS"`), or nil
- `YapperAPI:AddToDictionary(word)` → boolean (true if added; fires `SPELLCHECK_WORD_ADDED`)
- `YapperAPI:IgnoreWord(word)` → boolean (true if ignored; fires `SPELLCHECK_WORD_IGNORED`)

Example — check a word and get suggestions from another addon:

```lua
if YapperAPI and YapperAPI:IsSpellcheckEnabled() then
    if not YapperAPI:CheckWord("teh") then
        local suggestions = YapperAPI:GetSuggestions("teh")
        if suggestions then
            print("Did you mean:", table.concat(suggestions, ", "))
        end
    end
end
```

Post delegation
---------------
Post delegation lets another addon intercept messages that Yapper is about
to send, take responsibility for delivering them, and confirm delivery back
to Yapper.  This is useful for addons that want to transform, delay, or
route messages through their own systems.

### Protocol

1. Register a `PRE_DELIVER` filter that returns `false` to claim the message.
2. Yapper fires `POST_CLAIMED(handle, text, chatType, language, target)`.
3. The addon must call `YapperAPI:ResolvePost(handle)` within **5 seconds**
   to confirm it handled the delivery.
4. If the timeout expires without `ResolvePost`, Yapper sends the message
   itself and prints an error in the chat frame blaming the claiming addon.

### API

- `YapperAPI:ResolvePost(handle)` → boolean (true if the claim was valid)

### Example

```lua
-- Claim all outgoing messages, process them, then resolve.
local myHandle = nil

YapperAPI:RegisterFilter("PRE_DELIVER", function(p)
    -- Only claim SAY messages
    if p.chatType == "SAY" then
        return false   -- claim it
    end
    return p           -- let others pass through
end)

YapperAPI:RegisterCallback("POST_CLAIMED", function(handle, text, chatType, lang, target)
    -- Do custom processing...
    C_Timer.After(1, function()
        -- Tell Yapper we handled it.
        YapperAPI:ResolvePost(handle)
    end)
end)
```

### Blame

If an addon claims a message but fails to resolve within the timeout, Yapper
prints a chat-frame error identifying the addon by its registration source
(extracted from `debug.getinfo` at filter registration time).  An `API_ERROR`
callback is also emitted with kind `"delegation-timeout"`.

Notes for addon authors
-----------------------
- Filters must return the payload table (or `false`). Modify the table in-place.
- Prefer filters when you need to modify or cancel behavior; prefer callbacks
  for observational work.
- Do not assume registration order beyond numeric priority and registration time.
- If `_G.YapperAPI` is not present, your addon should guard or delay registration
  until Yapper has loaded.

Error reporting
---------------
When an external filter or callback errors, Yapper emits an `API_ERROR`
callback that addon authors can register for. This lets integrators inspect
the failure, log richer diagnostics, or attempt corrective actions (for
example, retrying a registration or sanitising payloads).

Signature: `function(kind, hook, handler_info, errorMessage, data, ...)`

• `kind` — string: `"filter"`, `"callback"`, or `"filter-return"` (unexpected return)
• `hook` — string: the hook or event name where the failure occurred
• `handler_info` — table or nil: `{ handle = <number>, priority = <number?> }`
• `errorMessage` — string: the error returned by the failing handler
• `data` — any: the payload table (for filters) or the returned value
• `...` — any: additional arguments that were passed to the handler

Example — register to log API errors:

```lua
YapperAPI:RegisterCallback("API_ERROR", function(kind, hook, handler_info, err, data, ...)
  print("Yapper API error:", kind, hook, handler_info and handler_info.handle, err)
end)
```

If no `API_ERROR` handlers are registered, Yapper falls back to emitting
the same concise diagnostic line via `YapperTable.Utils:DebugPrint` or
`print()` so failures remain visible during development.

Delivery semantics: Yapper will attempt to deliver the `API_ERROR` event
only to handlers registered by the same addon/module that owns the failing
handler (ownership is determined at registration time from the handler's
source). If one or more owner-specific `API_ERROR` handlers exist, only
those handlers are invoked. If none exist, the event is broadcast to all
registered `API_ERROR` handlers as a fallback.
